Green Campus Real Deployment Plan

Hardware, cost, and real-world integration guide

This document explains what you would need if you move the current Green Campus demo into a real campus installation. It covers hardware, rough India-focused cost ranges, deployment phases, and exactly how those devices would connect to the existing FastAPI + React project.

Fastest path Phase 1 pilot
Use existing meters, 2–4 smart energy meters, 1 edge gateway, and manual water/solar import where needed.
Best ROI Submeter high-value buildings first
Start with hostels, library, labs, admin, and main solar feed instead of instrumenting the full campus immediately.
Architecture Edge → API → dashboard
Hardware sends readings to an edge collector, the collector posts telemetry to the backend, and the dashboard uses the same summary logic it already has.
Budget band ₹70k to ₹6L+
Pilot deployments can stay under ₹1L. A serious multi-building rollout becomes a few lakh depending on sensor density and water instrumentation.

1. What hardware would be needed

The current project is software-ready for analysis, forecasting, comparison, and reporting, but it still depends mainly on imported Excel data. For real-time use, we need a data collection layer.

A. Electricity metering
Install RS485 / Modbus energy meters on each selected building incomer, main LT panel, or important subpanel.
  • Main campus incomer meter
  • One meter per major building or block
  • Optional AC / HVAC sub-metering later
B. Solar monitoring
Use inverter APIs, inverter Modbus registers, or a dedicated energy meter on the solar output line.
  • Best case: use the inverter's existing Modbus or vendor API
  • Fallback: meter the solar output line with a 3-phase energy meter
C. Water monitoring
For real-time water, use either pulse-output water meters or RS485 flow/level sensors for tanks and main branches.
  • Main incoming water line
  • Hostel and mess supply lines
  • Tank level sensors if direct flow meters are difficult
D. Waste data capture
Waste usually cannot be fully “real-time” like electricity. The practical approach is a hybrid system.
  • Digital weighing scale at collection points
  • Simple mobile/web entry form for housekeeping staff
  • Optional QR-tagged building-wise entry workflow
E. Edge gateway / collector
This is the local device that reads meters and sends cleaned readings to the backend.
  • Raspberry Pi 5 or small industrial PC for pilot
  • USB-RS485 converters or RS485 HAT
  • Optional LoRaWAN gateway if devices are spread out
F. Network and power
Each collector needs stable internet and backup power.
  • Campus LAN or 4G router
  • Mini UPS for collector
  • Panel-safe enclosure, wiring, and surge protection

  • 4 to 6 energy meters
  • 1 solar data source

  • Basic installation and network

Phase 2 ₹2L to ₹4L

4. How to integrate real hardware with this project

The current project already has a strong analytics layer. The missing real-time part is telemetry ingestion. So the right approach is not to rewrite the dashboard, but to add a device ingestion layer in front of the existing summary logic.

Step 1. Add a telemetry ingestion API
Create new backend routes like:
  • POST /ingest/energy
  • POST /ingest/water
  • POST /ingest/solar
  • POST /ingest/waste
Each route should accept device ID, building, timestamp, value, and quality/health fields.
Step 2. Add raw telemetry tables
Right now the project stores summarized-style records in tables like EnergyData and WaterData. For real-time scale, add raw telemetry tables first.
  • energy_telemetry_raw
  • water_telemetry_raw
  • solar_telemetry_raw
  • device_registry
Step 3. Keep the current dashboard models for rollups
Run a scheduled aggregation job every 5 minutes / 15 minutes / 1 hour to push cleaned totals into the current reporting tables.
  • This protects the existing dashboard logic.
  • It also keeps charts and forecasts fast.
Step 4. Build an edge collector service
The edge collector will sit near the hardware and do:
  • Modbus polling
  • retry and buffering during internet outage
  • local timestamping
  • secure API push to backend
Step 5. Add device health and missing-data monitoring
You should also store:
  • last seen time per device
  • battery/power status where possible
  • communication failure count
  • bad data / zero-value anomaly flags
Step 6. Extend UI with a device operations page
Add a new `Operations` or `Devices` view showing:
  • meter list
  • building mapping
  • last reading time
  • signal status
  • offline / stale devices

5. How each resource would be implemented

Electricity

  • Install RS485/Modbus meter on each selected building incomer.
  • Collector reads cumulative kWh and possibly voltage/current/power factor.
  • Backend stores raw readings and derives interval usage.
  • Existing dashboard endpoints continue to summarize by building and time window.

Solar

  • Preferred: read the solar inverter via Modbus or vendor API.
  • Fallback: install a dedicated solar export/production meter.
  • Map solar telemetry to the existing SolarData concept.

Water

  • Phase 1: use tank level and leak sensors plus manual meter reading import.
  • Phase 2: add pulse or RS485 water meters to main lines.
  • Phase 3: instrument hostel / mess / main branch lines for real per-building visibility.

Waste

  • Keep waste as operator-entered data rather than trying to force real-time automation immediately.
  • Use a staff tablet/phone form with building, waste type, and weight.
  • If needed later, integrate smart scales via API or local bridge.

6. Software work needed in this repo

Backend changes
  • Add device registry models and raw telemetry models
  • Add ingestion APIs with token-based authentication
  • Add aggregation jobs or scheduler
  • Add device health and stale-data alerts
  • Optionally add TimescaleDB / PostgreSQL partitioning later
Frontend changes
  • Add device operations page
  • Add live data freshness indicators
  • Add “last updated” badges for each building
  • Add ingestion health widgets and stale sensor alerts
  • Extend executive report with “data confidence” summary
Security changes
  • Per-device API keys or signed tokens
  • Role-based admin access for device setup
  • HTTPS-only ingestion
  • IP allow-list for edge devices if possible
  • Audit log for imported vs live-generated data

7. Best implementation approach for this exact project

Recommended Phase 1
  • Instrument 3 to 5 buildings only
  • Use existing solar data source if available
  • Keep water partly manual or tank-level based
  • Keep waste manual
  • Add one edge collector and one ingestion API
Recommended Phase 2
  • Add more submetering
  • Add leak alerts and better water sensing
  • Introduce device health dashboard
  • Start live alerting by email / WhatsApp / Teams
Recommended Phase 3
  • Move to near real-time forecasting
  • Add tariff optimization and peak-demand control
  • Add occupancy/event integration from timetable systems
  • Use the platform for real sustainability governance
Best practical recommendation If this becomes a real campus project, do not try to instrument everything at once. Start with one edge collector, a few RS485 energy meters, existing solar data, and simple water/waste workflows. That gives the fastest proof, the clearest ROI, and the safest path to a larger rollout.
